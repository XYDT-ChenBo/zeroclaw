# 检索与索引（RAG / 代码搜索）优化方案

本文档描述 ZeroClaw / zeroclaw 场景下「检索与索引」的质量与成本优化逻辑，并与增量一致性策略、全量重建时机、度量指标对齐。**不包含任何自动构建或 CI 执行步骤**；索引构建与全量重建的触发由维护者自行决定。

---

## 1. 目标与约束

### 1.1 质量目标

- 首轮召回尽可能覆盖**正确文件与符号**，减少「搜不到 → 多轮模型 / 工具试错」。
- 控制**误召回**与**排序靠后**，避免把噪声上下文塞进 LLM。

### 1.2 成本目标

- 降低单次查询的：embedding 调用次数、向量库往返、返回 chunk 数量与**合并后总字符数**（与后续 LLM token 强相关）。
- 降低索引侧：存储体积、增量维护成本；避免「检索省下的成本被上下文膨胀吃掉」。

### 1.3 工程约束

- 索引与检索 pipeline 需支持**版本化、可观测、可回滚**。
- **构建动作**：不在本文档中约定自动执行；何时执行增量构建、何时全量重建，由维护者在合适窗口人工或流水线触发。

---

## 2. 问题拆分：质量轴 vs 成本轴

| 维度 | 典型表现 | 主要抓手 |
|------|----------|----------|
| 召回质量 | 漏召、误召、排序差 | 分块、混合检索、融合重排、结果协议 |
| 检索成本 | 慢、重复扫、上下文爆炸 | Router 缩 scope、缓存、早停、增量索引、结果压缩 |

**原则**：先保证「能找对」，再在**不牺牲关键召回**的前提下压成本；若先粗暴压成本，易引发模型多轮补救，端到端成本反而上升。

---

## 3. 分层检索流水线（逻辑架构）

单次用户查询建议按阶段处理，每阶段有明确**输入 / 输出 / 早停条件 / 预算上限**：

1. **Router（意图与范围）**  
   判定：符号查找 vs 语义描述 vs 混合；输出仓库/路径白名单、最大候选数 `K`、是否跨仓。

2. **廉价精确通道（rg / 符号 / 路径）**  
   能精确命中的场景不依赖向量猜测；命中置信度高时可弱化或跳过语义检索。

3. **语义通道（Embedding）**  
   仅在 Router 缩小后的子集上检索；`topK` 宜小，靠融合与重排补全。

4. **结构通道（可选）**  
   调用关系、依赖图、最近变更模块等，用于「影响面 / 调用链」类问题。

5. **融合与重排**  
   多路结果合并（如 RRF、加权）；可选轻量重排模型或规则重排（路径/符号匹配加分、测试目录降权等）。

6. **结果压缩与打包（Context Packaging）**  
   输出结构化片段：路径、符号、行区间、短摘要、匹配原因；避免整文件灌入 LLM。

---

## 4. 混合检索逻辑概要

- **符号级**：标识符、路径、import 等——低歧义、低成本。  
- **语义级**：自然语言与错误片段——需控子集与 `K`。  
- **结构级**：图与依赖——适合链路与重构场景。  

**融合**：各路独立 `topK_i`，再融合为统一列表；按查询特征动态调权（例如查询中含明确符号则提高符号路权重）。

**二次检索（受控）**：仅在首轮结果明显不足或下游仍触发补搜时扩大 scope 或提高 `K`，并设**单次查询的二次检索次数上限**，避免默认双次全库扫描。

## 5. 索引生命周期与一致性

### 5.1 索引内容（质量上限）

- **分块**：优先按函数/类/语义块切分，块内保留路径、符号名、行号范围等元数据。  
- **元数据**：语言、模块、是否测试、最近变更等，用于过滤与重排。

### 5.2 增量更新逻辑（降本关键）

- **变更驱动**：以文件或 chunk 为粒度，仅对变更部分重算向量或局部索引条目。  
- **一致性标识**：  
  - 维护 **`index_version`**：索引整体或分片的逻辑版本（随分块策略、embedding 模型、schema 变更而递增）。  
  - 维护 **`content_hash`**：按文件或 chunk 的内容哈希，用于判断「工作区相对上次索引是否变化」。  

**查询时若发现工作区与索引不一致**，采用以下策略（本文档约定为**主策略**）：

- **仅对变更路径走实时 `rg`（或等价精确搜索）补洞**（中等成本）：  
  - 对已变更且尚未纳入增量索引更新的路径，不阻塞查询；用 **rg 在变更路径集合上**补充候选片段，与向量检索结果合并、去重后再重排。  
  - 未变更路径仍信任已有索引结果，避免全库重复 embedding。  

**说明**：该策略在「索引略旧」时仍能保持较高召回，成本介于「完全信任旧索引」与「全量实时重扫」之间；增量构建完成后，应对变更路径写回索引，逐步消除对 rg 补洞的依赖。

### 5.3 全量重建的时机（低频维护动作）

在逻辑上将 **全量重建** 定义为 **低频维护**，建议在以下情况考虑执行（**不自动执行**）：

- 分块策略、embedding 模型、向量维度、图谱 schema 等发生**不兼容变更**。  
- 索引损坏、校验失败或出现**无法通过增量修复**的大规模漂移。  
- 经度量确认检索质量长期不达标，且已排除 Router/重排/查询侧问题后，仍指向索引层缺陷。  

日常以 **增量更新 + 5.2 补洞策略** 为主；全量重建安排在维护窗口，由维护者决策。

## 5.4 方案落地点（最小改动实现）

为最小化对 `zeroclaw` 主流程的侵入，并提高封装性/独立性，本方案的「变更路径 rg 补洞」建议落在**检索工具层**，以“可选开关”形式启用：

- **工具**：`content_search`
- **开关**：`prefer_changed_paths=true`
- **manifest 路径**：`index_manifest_path`（默认 `state/retrieval-index.json`）
- **版本一致性**：可选 `expected_index_version`（当与 manifest 不一致时，将 manifest 中路径视为潜在陈旧候选，仍受 `changed_paths_limit` 约束）
- **变更路径上限**：`changed_paths_limit`（默认 500）

当启用该开关且检测到 workspace 相对 manifest 有变更时，`content_search` 会优先对“变更文件列表”执行 `rg`，达到「不全库重扫、仅补洞」的中等成本策略；当未检测到变更或 manifest 不存在时，行为与原先一致（对目录递归搜索）。

> 说明：该落地点满足单一职责/低耦合：\n+> - `retrieval/index_manifest` 仅负责一致性与变更检测；\n+> - `content_search` 仅负责把“搜索目标”从目录切换到文件列表；\n+> - agent/上层逻辑无需了解索引细节。

## 5.5 开关与配置矩阵（设计视角）

本节补充在落地 5.2/5.4 后涉及的**关键开关**、默认值与注意事项。仅描述设计与配置能力本身，不包含任何测试脚本、跑法或基准产物细节。

### 5.5.1 检索补洞开关（核心）

- **环境变量 `ZEROCLAW_RETRIEVAL_PATCH_ON_STALE`**
  - **默认**：无（取决于运行环境是否设置）；当未设置时不影响工具默认行为（仍以工具参数默认值为准）。
  - **作用**：允许 `content_search` 在 `prefer_changed_paths=true` 或环境触发时走“变更路径补洞”（manifest → changed paths → 对文件列表执行 `rg`）。
  - **影响面**：
    - 可能减少全目录递归搜索范围（更快、更省），但也可能因为 changed paths 选择或工具输出变长导致上下文膨胀（需配合 7.2 指标观察）。

- **工具参数 `content_search.prefer_changed_paths`**
  - **默认**：工具参数默认 `false`；当上层显式传入 `true` 或由环境策略触发时才启用。
  - **作用**：启用后尝试读取 manifest 并将搜索目标从“目录递归”切换为“变更文件列表”。

- **工具参数 `content_search.index_manifest_path`**
  - **默认**：`state/retrieval-index.json`（**相对 `workspace_dir`** 解析）。
  - **作用**：指定 manifest 位置；manifest 内路径为 workspace 相对路径（如 `src/agent/loop_.rs`）。

- **工具参数 `content_search.changed_paths_limit`**
  - **默认**：500
  - **作用**：变更文件列表上限，防止过大导致 `rg` 变成“近似全库扫描”。

### 5.5.2 Windows 依赖：`content_search` 的外部搜索程序

`content_search` 会优先使用 **ripgrep (`rg`)**。若找不到 `rg`，会退回到 `grep`。

- 在 Windows 环境中通常没有 GNU `grep`，因此**建议确保 `rg.exe` 在 PATH 中**。
- 若缺失，常见错误为：`Failed to execute search command: program not found`，会导致：
  - `content_search_calls` 增加但 `tool_calls_failed` 上升
  - 端到端 LLM 耗时上升（模型反复兜圈子补救）

### 5.5.3 是否可写入 `config.toml`？（配置文件项对照）

以下对照表回答“这些开关能否通过配置文件设置”。结论是：**一部分可以（如 observability/autonomy/agent）**，但**检索补洞策略本身（`ZEROCLAW_RETRIEVAL_PATCH_ON_STALE`）目前是环境变量级别开关**，并没有对应的 `config.toml` 字段。

| 开关/行为 | 可通过 `config.toml` 配置？ | 对应配置项（示例） | 默认与说明 |
|---|---|---|---|
| runtime trace 开关/路径 | ✅ | `[observability] runtime_trace_mode / runtime_trace_path / runtime_trace_max_entries` | schema 默认：`runtime_trace_mode="none"`、`runtime_trace_path="state/runtime-trace.jsonl"`、`runtime_trace_max_entries=200` |
| 交互审批/工具允许列表 | ✅ | `[autonomy] level / allowed_commands / auto_approve / always_ask / allowed_roots / non_cli_excluded_tools / require_approval_for_medium_risk / block_high_risk_commands` | 用于控制 shell 等工具的许可与审批策略；注意 shell 仍受 `allowed_commands` 约束 |
| agent 最大工具迭代数 | ✅ | `[agent] max_tool_iterations = 30` | schema 默认通常为 10；用于控制单轮允许的工具调用迭代上限 |
| provider 超时（LLM 网络超时） | ✅ | 顶层 `provider_timeout_secs = 300` | schema 默认 120；也可用环境变量 `ZEROCLAW_PROVIDER_TIMEOUT_SECS` 覆盖 |
| 检索补洞全局开关（Auto strategy C） | ❌（仅环境变量） | `ZEROCLAW_RETRIEVAL_PATCH_ON_STALE=1/0` | `content_search` 内部：若未显式传 `prefer_changed_paths`，则读取该环境变量作为默认 |
| 变更路径补洞的具体参数 | ❌（工具参数） | `content_search` 调用时传：`prefer_changed_paths/index_manifest_path/changed_paths_limit/expected_index_version` | 这是**工具调用参数**，不是全局配置字段；适合由上层策略（Router/agent）按场景决定 |

## 6. 结果压缩协议（面向 LLM）

检索层输出给下游时建议统一字段，例如：

- 路径 / URI  
- 符号（若有）  
- 行区间  
- `why_matched`（符号命中 / 语义相似 / rg 补洞 / 图邻居）  
- 短 `snippet`（已截断）  
- 置信度或分数  

目标：**先压缩再进模型**，使「检索结果合并后的总字符数」可控，并与第 7.2 节成本指标对齐。

## 7. 质量与成本的度量（没有指标就无法优化）

以下指标用于基线对比、回归发现与迭代验收。建议将其映射到 runtime trace / 观测系统的字段，以便长期可回归。

### 7.1 质量指标

| 指标 | 含义 | 用途 |
|------|------|------|
| **首轮是否命中正确文件/符号** | 人工标注或黄金集（golden set）上，Top-K 是否包含标准答案文件/符号 | 衡量召回与排序是否达标 |
| **需要二次检索的比例** | 单次用户任务中触发扩大 scope / 第二次检索的占比 | 过高说明 Router、融合或 `K` 需调整 |
| **下游工具 `file_read` 的平均次数与字节数** | 间接反映「检索是否够用」：次数多、字节大往往表示检索未覆盖关键点，模型被迫多读文件 | 与端到端 LLM 轮次联动分析 |

### 7.2 成本指标

| 指标 | 含义 | 用途 |
|------|------|------|
| **单次查询 embedding 次数** | 文本被编码为向量的调用次数 | 直接反映 embedding API / 本地编码成本 |
| **向量库 round-trip 次数** | 单次查询访问向量库的往返次数 | 发现多余重试或拆分查询 |
| **返回 chunk 数** | 融合重排后进入打包阶段的 chunk 数量 | 与上下文长度、噪声水平相关 |
| **索引存储体积** | 向量与元数据占用磁盘或对象存储 | 容量规划与压缩策略 |
| **增量构建耗时分布** | 单次增量任务 P50/P95 耗时 | 评估增量管线是否可接受 |
| **检索结果合并后的总字符数** | 进入 LLM 或工具链前的检索摘要总长度 | **与 LLM token 强相关**，核心成本代理指标 |

**验收建议**：任一优化上线前后，在相同黄金集上对比 7.1；在相同负载或采样查询上对比 7.2，并观察 `file_read` 与模型轮次是否下降。

## 8. 迭代节奏（逻辑顺序，不涉及执行）

1. 建立黄金集与 7.1 / 7.2 基线。  
2. 落地 Router + 混合检索 + 融合重排与结果协议。  
3. 实现 5.2：`index_version` + `content_hash`，不一致时 **变更路径 rg 补洞**。  
4. 持续增量索引，降低对 rg 补洞的依赖；仅在 5.3 场景安排全量重建。  
5. 按瓶颈单点迭代（误召、二次检索率、总字符数等）。
